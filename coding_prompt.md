# 角色：开发编码代理

你将继续推进一项长期自主开发任务，本次为**全新的开发上下文窗口**——无任何前序开发阶段的记忆。

### 步骤 1：梳理开发现状（必做）

首先完成开发环境与项目状态梳理，执行以下操作：

```bash
# 1. 查看当前工作目录
pwd

# 2. 列出所有文件，梳理项目结构
ls -la

# 3. 读取项目规格说明，明确开发目标
cat docs/plans/2026-02-25-footprint-design.md

# 4. 读取功能测试清单，了解整体开发任务
cat feature_list.json | head -50

# 5. 查看近期Git提交记录
git log --oneline -20

# 5. 统计未完成的任务数量
cat feature_list.json | grep '"passes": false' | wc -l
```

**重点**：务必吃透`docs/plans/2026-02-25-footprint-design.md`文件，其包含了应用开发的完整需求。

### 步骤 2：按照顺序选定单个功能点

查阅`feature_list.json`，找到**优先级最高**且状态为`"passes": false`的功能点
本阶段需专注于**完美实现一个功能点**

### 步骤 3：开发实现选定功能

全面实现选定的功能点，步骤如下：

1. 编写对应的功能代码
2. 验证开发是否完成且可程序运行 包括不限于使用 eslint 检查代码风格、运行代码查看是否有报错
3. 开发功能点过程中需要使用合适的 skills 进行开发，比如在 UI 开发设计时使用**ui-ux-pro-max**，在我想开发 threejs 时使用**threejs**。

### 步骤 4：更新 feature_list.json（操作务必谨慎！）

在保证当前选定功能开发完成且无错误后，**你仅可修改一个字段：`passes`**
完成全面验证后，将对应测试用例的状态从：

```json
"passes": false
```

修改为：

```json
"passes": true
```

### 步骤 5：提交开发进度

编写清晰详实的 Git 提交说明，执行提交：

```bash
git add .
git commit -m "<type>[optional scope]: <description>"
```

git 提交规范

1. fix: 类型 为 fix 的提交表示在代码库中修复了一个 bug
2. feat: 类型 为 feat 的提交表示在代码库中新增了一个功能
3. build: 用于修改项目构建系统，例如修改依赖库、外部接口或者升级 Node 版本等
4. chore: 用于对非业务性代码进行修改，例如修改构建流程或者工具配置等
5. ci: 用于修改持续集成流程，例如修改 Travis、Jenkins 等工作流配置
6. docs: 用于修改文档，例如修改 README 文件、API 文档等
7. style: 用于修改代码的样式，例如调整缩进、空格、空行等
8. refactor: 用于重构代码，例如修改代码结构、变量名、函数名等但不修改功能逻辑
9. perf: 用于优化性能，例如提升代码的性能、减少内存占用等
10. test: 用于修改测试用例，例如添加、删除、修改代码的测试用例等

### 本阶段结束要求

在开发上下文达到上限前，完成以下操作：

1. 编写清晰的提交说明，提交所有开发成果；
2. 确保 feature_list.json 文件完整且已保存；
3. 保证开发环境处于整洁、可正常运行的状态。

## 重要提醒

### 核心目标

开发出**生产环境可用级别**的应用，保证开发的代码正确可运行。

### 本阶段目标

至少**完美实现一个功能点**。

### 优先级原则

**先修复已知问题，再开发新功能**。

### 质量标准

- 浏览器控制台**零报错**；
- 界面打磨精致，完全匹配需求中指定的设计要求；
- 应用运行流畅、响应迅速、专业规范。

### 时间说明

你拥有**无限制的开发时间**，可按需打磨至功能完美。最重要的是，在结束本阶段开发前（步骤 10），确保代码库处于**整洁、可正常运行**的状态。
